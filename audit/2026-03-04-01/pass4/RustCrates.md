## Findings

### R12-P4-1 (LOW) -- Misleading CLI help text for `--context` argument

**File:** `crates/cli/src/commands/eval.rs`, lines 37-42

The `--context` argument help text says "key=value format, key is the context column name" but the actual parsing logic (lines 70-79) simply splits on commas and parses each part as a U256. There is no key=value parsing at all. The help text misleads users about how the argument works.

```rust
#[arg(
    short,
    long,
    help = "The context in key=value format, key is the context column name and value is the context rows as a comma separated list"
)]
pub context: Vec<String>,
```

Actual parsing:
```rust
let context = args
    .context
    .into_iter()
    .map(|ctx_str| {
        ctx_str
            .split(',')
            .map(|v| parse_int_or_hex(v).context("Invalid context value"))
            .collect::<Result<Vec<U256>>>()
    })
    .collect::<Result<Vec<Vec<U256>>>>()?;
```

### R13-P4-1 (LOW) -- `revm` version mismatch between workspace and eval wasm target

**File:** `Cargo.toml` (workspace root), line 18; `crates/eval/Cargo.toml`, line 22

The workspace defines `revm = "24.0.1"` but the eval crate's `cfg(target_family = "wasm")` dependency overrides it to `revm = "25.0.0"`. This is a full major version discrepancy. While this may be intentional (the wasm target may need a different revm version), it means non-wasm and wasm builds compile against different revm major versions, which risks semantic inconsistencies in EVM execution behavior between the two targets.

Workspace root:
```toml
revm = { version = "24.0.1", ... }
```

Eval crate wasm target:
```toml
revm = { version = "25.0.0", ... }
```

### R13-P4-2 (INFO) -- Inconsistent crate naming convention

**Files:** All `Cargo.toml` files

Crate names mix underscores and hyphens inconsistently:
- `rain_interpreter_dispair` (underscores)
- `rain_interpreter_bindings` (underscores)
- `rain_interpreter_parser` (underscores)
- `rain_interpreter_test_fixtures` (underscores)
- `rain-interpreter-eval` (hyphens)

Cargo normalizes hyphens to underscores for module paths, so this is not a functional issue, but it is a style inconsistency.

### R19-P4-1 (INFO) -- Near-duplicate `Parser2` trait definitions for wasm/non-wasm

**File:** `crates/parser/src/v2.rs`, lines 10-53 and 57-100

The `Parser2` trait is defined twice with the only difference being `+ Send` bounds on the non-wasm version. The trait bodies, method signatures, and default implementations are otherwise identical. This is a maintenance burden -- any change to one must be manually mirrored to the other.
