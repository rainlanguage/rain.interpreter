## Security Analysis

### Constructor Validation (lines 34-43)

The constructor enforces two invariants:
1. `opcodeFunctionPointers().length != 0` -- prevents zero-length table and consequent division-by-zero in `extern()`
2. `opcodeFunctionPointers().length == integrityFunctionPointers().length` -- ensures 1:1 correspondence between opcode and integrity functions

Both checks use raw byte lengths (not function-pointer counts). The odd-length edge case (A01-1 from prior audits) was dismissed as not user-controlled / generated by the build system.

### `extern()` Dispatch (lines 46-80)

- **Mod-wrapping** (line 76): `mod(opcode, fsCount)` bounds the opcode index. This is documented design (lines 55-65): cheaper than a bounds check, mirrors the main eval loop, prevents arbitrary memory read on out-of-range opcode. The tradeoff (silent wrap to a different valid opcode for direct callers) is acceptable because the integrity check provides parse-time safety.
- **Opcode extraction** (line 71): Masked to 16 bits via `& bytes32(uint256(type(uint16).max))`. Defense-in-depth; the `ExternDispatchV2` encoding only uses 32 bits total.
- **Operand extraction** (line 72): Low 16 bits of the dispatch. Correctly masked.
- **Function pointer load** (line 76): `shr(0xf0, mload(...))` extracts top 16 bits of 32-byte word, yielding the 2-byte function pointer. The `mload` may read past the array boundary for entries near the end, but the extra bytes are fully discarded by the shift.

### `externIntegrity()` Dispatch (lines 83-109)

- **Explicit bounds check** (lines 98-100): `if (opcode >= fsCount) revert ExternOpcodeOutOfRange(...)`. Correct for the integrity/parse-time path where precise error reporting matters.
- **Function pointer load** (line 105): Same `shr(0xf0, mload(...))` pattern as `extern()`. The bounds check on line 98 ensures the read is within the array.

### Assembly Memory Safety

Four assembly blocks (lines 68-70, 75-77, 94-96, 104-106). All are read-only: they compute pointer offsets or load values via `mload`. None write to memory or modify the free memory pointer. All correctly annotated `("memory-safe")`.

### Unchecked Arithmetic (lines 53-79, 90-108)

Both `extern()` and `externIntegrity()` run inside `unchecked` blocks. Operations:
- `fPointers.length / 2` -- division, cannot overflow
- `mod(opcode, fsCount)` -- result bounded by `fsCount`
- `mul(mod(opcode, fsCount), 2)` -- max value `2 * (fsCount - 1)`, where `fsCount` derives from a `bytes` length; cannot overflow
- `add(fPointersStart, ...)` -- pointer arithmetic within memory bounds

None can overflow or underflow.

### Reentrancy

`extern()` is `view`; `externIntegrity()` is `pure`. No state modifications. No reentrancy risk.

### ERC165 (lines 112-116)

Correctly overrides `supportsInterface` and chains to `super.supportsInterface(interfaceId)`. Reports support for `IInterpreterExternV4`, `IIntegrityToolingV1`, `IOpcodeToolingV1`, and `IERC165` (via super).

### Custom Errors

All three revert paths use custom errors: `ExternOpcodePointersEmpty` (line 37), `ExternPointersMismatch` (line 41), `ExternOpcodeOutOfRange` (line 99). No string-based reverts.

No findings. The contract has been thoroughly reviewed in prior audits (2026-02-17-03, 2026-03-01-01) and all previously identified issues have been triaged and either fixed or dismissed with valid rationale. No new security issues were identified in this review.
